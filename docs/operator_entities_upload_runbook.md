# Operator Entities Upload Runbook

## Current Contract

The normal operator flow is S3-first and save_path-only:

- workspace is created by name only;
- Beehive owns bucket/region/endpoint/prefix defaults;
- stages do not use `next_stage`;
- n8n routes outputs by returning manifest outputs with allowed `save_path`;
- entities are uploaded as JSON objects into the selected stage prefix.

Current defaults:

```text
bucket: steos-s3-data
region: ru-1
endpoint: https://s3.ru-1.storage.selcloud.ru
workspace_prefix: workspace name
```

## Create Workspace

1. Open `Workspaces`.
2. Enter `Workspace name`.
3. Click `Create workspace`.
4. Select the created workspace.

Do not enter S3 bucket, endpoint, region, prefix, workdir, pipeline path, or database path. Those are generated by the backend.

Rejected names:

- empty;
- spaces only;
- names containing `/` or `\`;
- names containing `..`;
- names with control characters.

## Create Stage

1. Open the workspace.
2. Go to `Stage Editor`.
3. Enter:
   - `Stage ID`;
   - production n8n webhook URL;
   - max attempts;
   - retry delay;
   - `Terminal stage` if the workflow may finish without output artifacts.
4. Click `Create S3 stage`.
5. Copy generated save_path aliases when configuring n8n outputs.

Do not configure `next_stage`. It is deprecated. n8n must choose the output route by manifest `save_path`.

## Upload Entities

1. Open workspace `Entities`.
2. Choose `Target stage`.
3. Click the folder file input.
4. Select a local folder containing `.json` files.
5. Wait for the import summary.

Only JSON objects are accepted. Arrays, strings, numbers, booleans, and `null` are reported as invalid per file.

Upload destination:

```text
s3://steos-s3-data/{workspace_prefix}/stages/{stage_id}/{file_name}
```

Beehive uploads with metadata:

```text
beehive-entity-id
beehive-artifact-id
beehive-stage-id
```

After upload, Beehive registers the artifact as pending on the selected stage. A separate S3 reconcile is not required for these uploaded files.

## Entity Operations

Use `Entities` for normal work:

- search/filter by stage/status;
- select rows;
- archive entity;
- restore archived entity with `Show archived`;
- open details;
- run selected pipeline waves.

Archive does not delete S3 objects. It only hides the entity from the default list and marks it archived in SQLite.

Entity edit is intentionally limited to:

- display name;
- operator note.

Business JSON editing is not part of B9.

## Run Selected Pipeline Waves

1. Select one or more visible non-archived entities.
2. Click `Run selected pipeline waves`.
3. Check row status and entity detail/run outputs.

The run requires a real reachable n8n production webhook. If a placeholder webhook is configured, Beehive can still claim the selected artifact, but the task will fail at HTTP execution.

## Diagnostics

Use Advanced/Diagnostics only when needed:

- raw validation issues;
- YAML preview;
- route internals;
- detailed counters;
- legacy link diagnostics.

Normal operators should not need to edit S3 route internals or server paths.

## Smoke Command

The B9 smoke uses a temporary registry/root and real S3 upload:

```bash
mkdir -p /tmp/beehive-b9-entities-smoke
BEEHIVE_WORKSPACES_CONFIG=/tmp/beehive-b9-entities-smoke/workspaces.yaml \
BEEHIVE_WORKSPACES_ROOT=/tmp/beehive-b9-entities-smoke/workspaces \
BEEHIVE_SERVER_PORT=8787 \
cargo run --manifest-path src-tauri/Cargo.toml --bin beehive-server
```

In another shell:

```bash
BEEHIVE_API_BASE_URL=http://127.0.0.1:8787 node scripts/web_operator_entities_smoke.mjs
```

Expected high-level result:

- workspace created by name only;
- one stage created with `next_stage = null`;
- deprecated `next-stage` route returns `next_stage_deprecated`;
- three JSON objects uploaded to S3;
- entities listed;
- one entity archived and restored;
- selected-run route responds;
- stage and workspace are archived/deleted from the temporary registry.
